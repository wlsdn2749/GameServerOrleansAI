# AGENTS.md — GameServerOrleansAI

에이전틱 코딩 작업 지침 및 프로젝트 누적 문서. 모든 에이전트는 작업 전 이 문서를 읽고,
의미 있는 결정/구조 변경 후 이 문서를 갱신한다.

## 프로젝트 개요
- **목표**: Microsoft Orleans(Actor Model) 기반 MMO 월드 게임서버.
- **전송**: TCP + 커스텀 바이너리 프로토콜.
- **영속화**: In-Memory(개발용) → 추후 Redis/SQL.

## 개발 방법론 (필수)
- **TDD (Red → Green → Refactor)**: 새 grain/protocol/도메인 로직은 **실패하는 xUnit 테스트를 먼저** 작성하고,
  통과시키는 최소 구현 후 리팩터.
- 테스트: **xUnit** + Orleans **`Microsoft.Orleans.TestingHost`**.
- `dotnet build`와 `dotnet test`가 **그린**일 때만 PR 병합.

## Git / PR 워크플로우
- main 직접 푸시 금지 (최초 부트스트랩 제외).
- 기능 단위 feature 브랜치 → PR → **스쿼시 병합**.
- 브랜치 명명: `feat/<slug>`, `fix/<slug>`, `chore/<slug>`.
- PR은 "빌드+테스트 통과하는 동작 슬라이스" 단위.
- 커밋/PR 본문 말미에 `Co-Authored-By: Claude` 서명 포함.
- **리뷰 게이트 (RED/BLUE 2-에이전트 만장일치)**: PR을 올린 직후 **독립 서브에이전트 2명**이 diff를 검토한다.
  각 리뷰는 **"왜(Why)"와 "어떻게(How)"** 에 집중한다 — 이 결정이 *향후 어떻게 평가될지*, *어떻게 업데이트되면 좋을지*.
  - **RED — 적대적 검토(attacker mindset)**: 악의적 클라이언트/적대적 입력, 경쟁 상태, 스케일아웃(멀티 사일로/게이트웨이), 남용·DoS·상태 오염·보안 구멍을 적극적으로 찾는다. "어떻게 깨지고 악용되는가, 왜 위험한가"를 논증.
  - **BLUE — 보수적 검토(maintainer mindset)**: 장기 유지보수성, 정상 동작 정확성, Orleans 베스트프랙티스, API/테스트 설계, 진화 용이성. "이 설계가 향후 어떻게 평가될까, 어떻게 고쳐 두면 좋을까"를 논증.
  - 각 리뷰는 마지막에 명시적 평결 `APPROVE` 또는 `REQUEST_CHANGES`를 남긴다.
  - **만장일치 규칙**: RED·BLUE 둘 다 `APPROVE`일 때만 병합 허용. 하나라도 `REQUEST_CHANGES`면 같은 브랜치에서 수정 → **두 에이전트 모두 재검토** → 만장일치까지 반복.
  - 빌드+테스트 그린은 기본 전제(이와 별개로 항상 충족).

## 솔루션 구조
```
src/GameServer.Protocol       # opcode, 패킷 record, 프레이밍, MemoryPack
src/GameServer.Abstractions   # grain 인터페이스, 스트림 이벤트 DTO
src/GameServer.Grains         # grain 구현
src/GameServer.Silo           # Orleans 호스트
src/GameServer.Gateway        # TCP 리스너 + Orleans 클라이언트 + 세션
tools/GameServer.TestClient   # 스모크 테스트 콘솔 클라이언트
tests/GameServer.Tests        # xUnit + TestingHost
```

## 프로토콜 프레이밍 규약
```
[ length: int32 LE ][ opcode: uint16 LE ][ payload: MemoryPack bytes ]
length = opcode(2) + payload 길이
```
- opcode 정의는 `GameServer.Protocol/Opcode.cs`의 단일 enum에 집중.
- 패킷 DTO는 `[MemoryPackable] partial record`.

## 코드 스타일
- Nullable enable, file-scoped namespace, ImplicitUsings.
- async 메서드는 `Async` 접미사(공개 grain 인터페이스는 Orleans 관례상 생략 가능).
